# AGENTS.md — Dietician

Conventions + workflows for any LLM agent (Claude Code, jarvis, future) working on this codebase.

## Project type

Personal Dietician for one user (Victor, UAIC year-1 AI student in Iași, RO). Kotlin Multiplatform Compose (Android + Windows Desktop). VPS-canonical merge replica via Tailscale mesh on user's existing ByteHosting VPS. Mergeable into `jarvis-kotlin` life-OS later as a Subsystem.

**Spec is source of truth:** `docs/superpowers/specs/2026-05-17-dietician-design.md`. Read it before any non-trivial change.

## Three-layer LLM-Wiki pattern (per the Wiki essay)

1. **Raw sources** (`raw/`) — immutable inputs the user/agent drop in (recipe URLs, flyer PDFs, receipts, voice notes, textbooks, papers, datasets, LLM raw responses).
2. **Wiki** (`wiki/`) — LLM-maintained markdown. Narrative + interlinked. Two-file pattern for any page with live data: `slug.md` (user-editable narrative, no numbers) + `slug.data.md` (autogen, header `<!-- AUTOGENERATED. EDITS OVERWRITTEN. -->`). Narrative transcludes data via Obsidian `![[slug.data]]`.
3. **Schema** = this file + the spec.

## Storage decision rule (load-bearing)

- **Postgres on VPS + SQLite per-client:** structured/numeric/transactional state (prices, pantry events, weight events, macros, budgets, baselines).
- **Markdown wiki:** narrative reasoning, learned heuristics, rationale, methodology pages, summaries, "applied to you" sections.
- **`raw/`:** immutable inputs too large or unstructured for parsed ingestion. Indexed (filename + abstract + section TOC in SQL/wiki) for on-demand retrieval.

**NEVER put numbers in markdown bodies.** Use transclusion from `.data.md` files. Numbers live in SQL.

## Local-first writes + event-sourced ledger (Council 4 BREAK fix)

Each client (phone, desktop) is the **canonical writer** for events it originates. VPS Postgres = merged read replica + cross-device coordinator.

Inventory + meal + weight + receipt = append-only event log:
- `pantry_events(event_uuid, device_id, originated_at, synced_at, sku_uuid, delta_qty, unit, reason, evidence_ref)`
- Pantry "current quantity" = `SUM(delta_qty) OVER (sku_uuid)`. No LWW.
- Metadata (food name, expiry, open-status) uses LWW per row — that's an assignment, not a delta.

Sync = outbox (per-client SQLite) → push to VPS `/sync/push` → VPS merges → broadcast via ntfy + WebSocket.

## Ingest operation

When user drops a new source into `raw/`:
1. Read source
2. Discuss key takeaways with user
3. Write summary page (with frontmatter, citations)
4. Update `wiki/index.md`
5. Update entity/concept pages that reference this topic (5-15 page touches typical)
6. Cross-link via embedding similarity (pgvector top-5 candidates, single LLM confirm)
7. Append `wiki/log.md` entry: `## [YYYY-MM-DD] ingest | <Source Title>` (parseable: `grep "^## \[" log.md`)

## Query operation

When user asks a question against the wiki:
1. Search relevant pages via index + BM25 + embeddings hybrid (Lucene + pgvector + LLM rerank top-20)
2. Read top pages, synthesize answer with **citation per quantitative claim** in format `[per slug.md#section-anchor: "..."]`
3. If answer derived new heuristic / table / rule reusing >2 sources → propose new wiki page (preview to user, save on confirm) tagged `authority: derived`

## Lint operation

Periodically (weekly cron):
1. Contradictions between pages
2. Stale claims (`last_verified > review_cadence_days`)
3. Orphan pages with no inbound links
4. Concepts mentioned but lacking own page
5. Missing cross-references
6. Data gaps fillable via web search

## LLM provider routing (must respect)

- All LLM calls go through `:shared:llm.LlmRouter`
- Vision: ClaudeMax CLI on desktop primary → OpenRouter Gemini 2.0 Flash Vision fallback
- Text (hard): OpenRouter `anthropic/claude-3.5-sonnet` primary → ClaudeMax CLI fallback
- Text (mechanical): OpenRouter `google/gemini-2.0-flash-exp` → `anthropic/claude-3.5-haiku`
- Embeddings: OpenRouter `voyage/voyage-3-lite` → local Ollama `nomic-embed-text`
- Whisper: local `whisper.cpp large-v3-turbo` ONLY

**Two-phase budget reserve** required for every call. **Idempotency keys** persisted pre-call. **`@RequiresNoActiveTransaction` guard** at every provider entry-point (no LLM calls inside DB transactions).

## Vision JSON corruption gate (load-bearing)

Every Vision call MUST:
1. Persist raw response to `state/llm-raw/{call_uuid}.txt` BEFORE parse
2. Strict-mode parser: strip fences, JSON-schema validate
3. On parse failure: log + skip + queue for review, NEVER partial-extract
4. `price_observations` row with `extraction_call_id` + `raw_evidence_ref`
5. `price_posterior` recomputation EXCLUDES `source='llm:vision'` UNLESS confirmed by Monitorul OR Playwright within ±7d

## Refusal triggers (LLM prompt mandate)

See spec §28. Hardcoded refusals for ED-risk patterns, unrealistic weight-loss requests, medical-advice asks, fasts >36h. Never gamify weight loss. Never overstep clinical role.

## Anti-patterns (do not introduce)

- ✘ Numbers in markdown body (use `.data.md` transclusion)
- ✘ LWW conflict on inventory deltas (use event-sourced SUM)
- ✘ Periodic WorkManager (event-driven only)
- ✘ Tailscale self-report as health signal (use app-layer `/health` ping)
- ✘ Telegram as user interface (in-app Compose UI; ntfy for push only)
- ✘ Playwright in-process (subprocess JAR with RSS ceiling)
- ✘ Stringify structured data into `SubsystemOutput.text` (use `state/meal-plan-current.json` side-channel)
- ✘ Wiki regenerate that clobbers user narrative edits (file-watcher suppresses)
- ✘ Vision price write into baseline without cross-source confirm
- ✘ Speculative jarvis-merge driving Dietician architecture decisions

## 🔴 Devil's Advocate
AGENT: Devil's Advocate
STANCE: REJECT
REASONING: Step 7 is a ticking time bomb — deriving `pantry_current` as a live `SUM(delta_qty) GROUP BY item HAVING > 0` Flow over an append-only ledger means every pantry read scans the entire event history forever, and every pantry-screen recomposition re-scans it. At ~10 deltas/day for a single user that's tolerable for six months and catastrophic at year three, exactly when migration cost is highest and you've forgotten why the UI got slow. The council locked "event-sourced ledger" as the write model, not as the read model — you need a materialized `pantry_current` snapshot table updated in the same transaction as the ledger insert (or a periodic compaction checkpoint with delta-since-checkpoint summing), not a naive view. Also: step 4's "outbox insert in same transaction as event write" is correct, but step 9 has zero test for the failure mode where the outbox drain succeeds on the server but the local `acked` flip fails (process crash between HTTP 200 and DB write) — that's the canonical at-least-once duplicate vector and it's not in the chaos test list.
KEY CONCERN: Unmaterialized `pantry_current` view over an unbounded append-only ledger — O(history) per read, guaranteed degradation, no compaction story in plan.

## 📚 Domain Expert
AGENT: Domain Expert
STANCE: CONDITIONAL
REASONING: The architecture is textbook: transactional outbox + event-sourced ledger with idempotent UPSERT-by-UUID + pull-since cursor is the canonical offline-first stack (cf. CouchDB replication, Litestream, Local-First Software by Kleppmann et al. 2019). However, two divergences from convention lack justification: (1) ordering the outbox drain by `originated_at` ASC is wall-clock-causal, not logically-causal — established prior art (Lamport 1978, HLC by Kulkarni 2014) uses hybrid logical clocks or per-aggregate sequence numbers. (2) `pull_trigger` debounce of 200ms with no per-table cursor advancement guarantee risks the classic "lost-update on cursor" trap where `last_sync_at` advances past an event that arrived mid-pull (cf. Debezium snapshot+streaming handoff). LWW-on-tie-prefer-remote for `pantry_metadata` is fine for single-user but should be explicitly marked as such. The `SELECT … HAVING SUM(delta_qty) > 0` materialized-on-read pattern is sound but needs a snapshot/checkpoint story before the ledger grows past ~10k events.
KEY CONCERN: Outbox drain ordered by `originated_at` (wall clock) instead of a monotonic per-client sequence or HLC — client clock skew or NTP jumps can violate causal order on replay.

## ⚙️ Pragmatist
AGENT: Pragmatist
STANCE: CONDITIONAL
REASONING: For n=1 user the event-sourced ledger + outbox + WS + ntfy + pull-since + LWW stack is genuinely overbuilt, but it's spec-locked so the pragmatic ask is to keep the debug surface survivable rather than re-litigate scope. The real 2am-pain comes from having four trigger paths (WS push, ntfy push, pull-since timer, manual `/diag` retry) all feeding one debounced channel — when sync misbehaves you won't know which path fired or which dropped. Drain-worker "gives up after 10 attempts and surfaces in /diag" is fine only if `/diag` actually shows the event payload + last error + which trigger path was active, otherwise dead-letter events become silent data loss. The `pantry_current` Flow doing `SUM(delta_qty) HAVING > 0` over an unbounded append-only ledger is correct day-one and a perf cliff day-365 with no compaction story.
KEY CONCERN: Add a single structured `sync_log` table (trigger source, event_uuid, attempt#, outcome, error) written by every sync path, queryable from `/diag` — without it, debugging the four-way trigger fan-in at 2am is impossible.

## 🧱 First Principles
AGENT: First Principles
STANCE: CONDITIONAL
REASONING: With 1 user and 2 devices, most of this is genuinely required because the physics of "offline phone + offline desktop + canonical server" force idempotency, atomicity, and an outbox regardless of team size — event-sourcing is also load-bearing for audit/replay. However, several pieces are team-scale lore: LWW is solving a problem that barely exists when one human edits one row at a time, 10-attempt exponential backoff with batches of 50 is sized for fleet throughput not a 2-device trickle, and a dedicated `metadata/` package for LWW is architectural ceremony around a 5-line `MAX(updated_at)` rule. The WS + ntfy dual-channel — ntfy push already wakes the client; the WS adds a second mechanism solving the same "tell me to pull" problem, doubling failure surface for one user who tolerates 200ms-30s latency. The `pantry_current` SUM-view is correct first-principles (derived state from the ledger, never stored) and the same-tx event+outbox write is the one non-negotiable atomicity invariant.
KEY CONCERN: Dual push channels (WS + ntfy) and a `metadata/` LWW subsystem are solving distributed-systems problems that don't exist at n=1 user / n=2 devices — collapse to ntfy-only + inline `MAX(updated_at)` unless a concrete single-user scenario justifies each.

## ⚠️ Risk Analyst
AGENT: Risk Analyst
STANCE: CONDITIONAL
REASONING: Catalog by severity: (1) CATASTROPHIC — LWW on `pantry_metadata` with wall-clock from phone/desktop/VPS will silently corrupt under clock skew (phone NTP drift, desktop sleep wake, VM time-jump); a phone with a 30s-fast clock permanently wins every tie and you won't notice until inventory goes negative. Use Lamport/HLC or server-stamped `received_at`, not device `last_modified_at`. (2) CATASTROPHIC — `event_uuid` as idempotency key on event-sourced tables only works if the client generates the UUID BEFORE the outbox insert AND reuses it on retry; if the drain regenerates UUIDs on retry-after-crash, duplicates inflate `SUM(delta_qty)`. UUID must be column-default at event-table insert time, same tx as outbox. (3) HIGH — pull-since with `last_sync_at = max(server_ts)` has the classic boundary-race: events with ts == last_sync_at committed AFTER your pull are lost forever; must use `>= last_sync_at` + dedupe by uuid, or server returns a cursor/LSN not a timestamp. (4) HIGH — Flyway migrations in `:server/resources` with no parallel SQLDelight migration discipline means schema drift is inevitable; need a single source of truth + CI check. (5) MEDIUM — WAL + Android Doze: `busy_timeout=5000` won't save you if Doze suspends mid-`BEGIN IMMEDIATE`; WAL checkpoint can be deferred indefinitely → `-wal` grows unbounded → ENOSPC. Need `wal_autocheckpoint` + foreground checkpoint on app-background. (6) MEDIUM — outbox 10-attempts-max then what? Silent drop = data loss; must move to dead-letter table surfaced in `/diag`, never delete. (7) MEDIUM — `pantry_current` as live view over `SUM(delta_qty)` is O(n) per query; after 6 months of events this is a perf cliff. (8) LOW-but-irreversible — schema in `:shared` means changing event shape later requires coordinated phone+desktop+server release.
KEY CONCERN: LWW with device wall-clock on `pantry_metadata` + non-deterministic `event_uuid` regeneration on retry — together silently corrupt inventory in ways the chaos/replay tests won't catch unless they explicitly simulate clock skew and mid-tx crash with outbox-row-missing-uuid.

SANITY Domain Expert: FLAG
NOTE: The Debezium analogy is misapplied. Debezium uses LSN watermarks because it consumes a shared write-ahead log on a single server with concurrent writers; the "lost-update on cursor" trap there is about WAL position ordering under concurrent transactions, not about a pull-since-timestamp client. The actual `last_sync_at` boundary-race risk is real and Agent 5 catches it correctly with the `>=` + dedupe framing — but invoking Debezium snapshot+streaming handoff inflates the precedent. Also, "wall-clock-causal" is loose terminology: `originated_at` ASC on the client's own outbox is per-client monotonic if generated from a single clock source on insert, so HLC/Lamport is overkill for ordering within one client's outbox drain — it only matters for cross-client merge on the server, which the proposal handles via UPSERT-by-UUID, not by ordering.

SANITY Pragmatist: PASS
NOTE: reasoning holds

SANITY First Principles: FLAG
NOTE: The "ntfy already wakes the client, WS is redundant" claim misreads the two channels' roles. ntfy is a push-from-server notification mechanism that works when the device is backgrounded/Doze'd, whereas WS is a foreground-app bidirectional channel for sub-second sync while the app is open. They cover disjoint device states (foreground-open vs backgrounded), not the same problem — collapsing to "ntfy-only" would push every foreground sync through the higher-latency push infra. The "doubles failure surface" framing treats them as redundant when they're complementary. The LWW/metadata critique stands; the dual-channel critique does not.

## Judge

VERDICT: FLAWED
CONFIDENCE: 8

SUMMARY:
The architecture is sound — event-sourced ledger + outbox + UPSERT-by-UUID + per-client SQLite is the right shape for a Tailscale-canonical multi-device system. But Plan-1 cannot ship as-described: eight convergent defects (perf cliff, clock-skew LWW, boundary race, schema drift, WAL+Doze, dead-letter, sync_log, ack-vs-flip duplicate) survived sanity and each maps to a concrete artifact the plan must produce, not "consider later."

REQUIRED CHANGES (if FLAWED):
- `pantry_current` perf cliff → materialize as SQLDelight table maintained by trigger or app-side fold on event insert; ship a compaction job (rollup snapshot per item every N events or T days) with a task in Plan-1 named "pantry compaction"; include a benchmark task asserting query stays <10ms at 100k events.
- LWW with device wall-clock = silent corruption → use a hybrid (server_recv_at, device_monotonic_seq, device_id) tuple as the LWW key; never trust raw `System.currentTimeMillis()` for conflict resolution; add Plan-1 task "metadata conflict resolver" with property test injecting ±24h clock skew.
- Pull-since boundary race → use `>` strictly with a half-open `(last_sync_at, now]` window AND tiebreak by `event_uuid` ordering; cursor is `(timestamp, uuid)` not `timestamp` alone; add property test "no event dropped, no event double-applied across 10k randomized pulls."
- Flyway/SQLDelight schema drift → single CI task "schema parity": Flyway migrate against ephemeral Postgres, dump schema, normalize, diff against generated SQLDelight schema dump; CI fails on diff; this is a Plan-1 task, not a follow-up.
- WAL + Doze checkpoint discipline → explicit `PRAGMA wal_autocheckpoint` setting + `PRAGMA wal_checkpoint(TRUNCATE)` on app background AND on WorkManager periodic job (15min); document Doze interaction; add chaos test killing process mid-WAL.
- Outbox dead-letter table → `outbox_dead` table with full payload + last_error + attempt_count + first_failed_at; surfaced in /diag with manual-replay endpoint; NEVER silent-drop after 10 attempts; Plan-1 task explicit.
- `sync_log` table for 4-way trigger fan-in → row per trigger fire: (source: WS|ntfy|manual|periodic, fired_at, debounced_to, pull_started_at, pull_ended_at, events_pulled, error); /diag reads this; without it the 4-way fan-in is undebuggable.
- "Server-acked but local-flip-failed" duplicate vector → explicit chaos test: kill process between `server.ack(event_uuid)` returning 200 and local `outbox.delete(row_id)` committing; assert next drain does NOT re-send (idempotency check: query server `events WHERE event_uuid = ?` before resend OR rely on server-side UPSERT-by-UUID being truly idempotent and document the invariant).

RECOMMENDED PATH:
Accept the architecture; tighten Plan-1 with the eight tasks above before any impl agent touches code. Each fix is a concrete file + test, not a discussion item. The plan should produce: SQLDelight `.sq` files, Flyway migrations, a `sync_log` + `outbox_dead` table, a compaction job, a schema-parity CI gate, and a chaos suite covering clock-skew + boundary race + ack-vs-flip + WAL-kill. Defer nothing on these — they are correctness-critical, not polish. Everything else in the approach (module layout, WS+ntfy duality, UUID-at-insert, 200ms debounce, Ktor/SQLDelight choice) ships as written.

NOTES ON DISMISSED CONCERNS:
Ignore the Debezium LSN / HLC requirement (per-client outbox is monotonic on one clock; cross-client merge is UPSERT-by-UUID), the WS+ntfy collapse (they serve different power/foreground modes), and the event_uuid regen risk (design generates UUID + outbox row in the same tx, so the bug requires inventing a violation that isn't there).
